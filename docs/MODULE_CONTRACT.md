# عقد الوحدات الموحد (مسودة — تُعتمد نهائيًا في الخطوة 13)

كل وحدة في النظام (محرك القافية، محلل الوزن، خريطة الفراغات، طبقة LLM...) تلتزم الشكل التالي قبل كتابة أول سطر فيها.

## 1. الواجهة
```ts
interface ModuleResult<T> {
  ok: boolean;
  data: T | null;          // ناتج الوحدة وفق schema معلن
  error: ModuleError | null;
  meta: { module: string; version: string; took_ms: number };
}

interface ModuleError {
  code: string;            // ثابت قابل للبرمجة (مثال: "TAQTEE_FAILED")
  message_ar: string;      // رسالة عربية تصلح للعرض المباشر
  recoverable: boolean;    // هل يمكن إعادة المحاولة؟
}
```

## 2. القواعد
1. **مدخلات ومخرجات معلنة:** لكل وحدة ملف schema يعرّف مدخلاتها ومخرجاتها قبل تنفيذها.
2. **لا وحدة تعدل بيانات وحدة أخرى:** القراءة مشتركة، والكتابة حصرية للمالك المعلن (مثال: Beat Profile يكتبه المعمل فقط؛ تصحيحات المستخدم في وثيقة overrides منفصلة).
3. **الفشل صريح لا صامت:** أي خطأ يرجع ModuleError كاملًا — لا null غامض ولا استثناء غير ملتقط.
4. **العربية أولًا في الرسائل:** كل ما قد يظهر للمستخدم يُكتب بالعربية من المصدر.
5. **قابلية التبديل:** الوحدات ذات المزودين المتعددين (LLM) تلتزم واجهة مجردة واحدة، والاختيار بمتغير إعداد.

## 3. التسمية
- وحدات: `rhyme-engine`, `weight-engine`, `slot-mapper`, `beat-lab`, `llm-provider`, `coach`
- دوال التحليل ترجع دائمًا «بصمة» JSON قابلة للتخزين مع الكيان الذي حللته.
